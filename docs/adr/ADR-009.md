# ADR-009: LLM Provider Abstraction & Fallbacks

## Status
Accepted

## Context
Relying on a single LLM provider exposes Rover to downtime, rate limits (`RESOURCE_EXHAUSTED` errors), or cost constraints. We need a unified client interface that defaults to highly performant models but falls back to alternatives seamlessly.

## Decision
Create an LLM client abstraction layer in `src/llm.py` that defaults to Google's Gemini SDK (for high-context and cost-efficient structured output) and automatically catches rate limits to reroute requests to OpenRouter (e.g. DeepSeek v3).

## Consequences
Positive:
- Robust agent execution: Rover will continue to function even if a developer's Gemini quota is exhausted.
- Unified response model: The system maps different vendor schemas into a single Pydantic output.
- Future-proof: Switching models or adding vendors only requires updating `src/llm.py`.

Negative:
- Running fallbacks through third-party routers like OpenRouter requires maintaining secondary API key setups.
