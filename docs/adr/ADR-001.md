# ADR-001: Why Gemini Function Calling over LangChain

## Status
Accepted

## Context
Rover needs a reasoning loop where a language model decides which Python
functions to call, in which order, based on what it reads from the
codebase. Two approaches were considered:

1. Use an agent framework (LangChain, LangGraph) that wraps orchestration
   in a library.
2. Use the Google Gen AI SDK's native function calling API directly,
   with a raw Python while loop for orchestration.

## Decision
Use the Google Gen AI SDK (google-genai) function calling API directly.
No agent framework is used.

## Consequences

Positive:
- Full control over the while loop. Logging, error handling, and retry
  logic go exactly where I need them.
- When something fails, the traceback points to my own code, not
  framework internals.
- Zero extra dependencies beyond google-genai.
- I understand every line because I wrote it. This matters for
  interviews and mentor 1:1 discussions.
- Gemini 2.5 Flash has a generous free tier compared to GPT-4o,
  making development practical without credit concerns.

Negative:
- Gemini's function calling response format (response.function_calls,
  types.Part.from_function_response) is different from OpenAI's format.
  Required learning the google-genai types module.
- The message history format (types.Content with parts) is more verbose
  than OpenAI's simple role/content dicts.
- If the project grows to multiple parallel agents, a framework like
  LangGraph would be the right choice.

## Alternatives Considered

LangChain: Abstracts the API differences between providers, which
would have made the OpenAI-to-Gemini switch easier. However, it adds
abstraction layers that make debugging harder at this stage. When a
tool call fails unexpectedly, raw SDK errors are much clearer than
LangChain-wrapped errors.

LangGraph: Purpose-built for stateful agent loops. Worth revisiting
when Rover grows to multiple agents with shared state.

GPT-4o (OpenAI): Evaluated initially. Switched to Gemini 2.5 Flash
for the free tier, which removes API cost as a constraint during
development and testing.
