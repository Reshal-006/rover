# ADR-003: Why scan results are persisted as JSON files

## Status
Accepted

## Context
The new proactive scan workflow needs a way to store repository findings, scan metadata, and bug statuses. That data should be accessible to both the dashboard and API endpoints.

## Decision
Persist scan results as JSON files in a local `scans/` directory. Use a simple storage layer to save, load, and list scan records.

## Consequences
Positive:
- Minimal additional dependency footprint.
- Easy inspection and debugging by opening the JSON files directly.
- Simple persistence that can later migrate to SQLite or another database.
- Works across development and deployed environments without special setup.

Negative:
- Not as powerful as a relational database for queries.
- JSON file contention could be an issue if many scans are written concurrently.

## Alternatives Considered
- SQLite: a stronger long-term choice, but adds complexity for this first version.
- No persistence: would make the dashboard and scan APIs stateless and force users to rescan every time.

## Decision Rationale
JSON persistence is the right entry point for this stage. It allows the dashboard to surface scan history immediately while leaving room to upgrade storage later. The `ScanStore` abstraction encapsulates the storage format so migration is straightforward.
