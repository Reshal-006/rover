# ADR-002: Why the proactive scan workflow uses a lightweight scanner

## Status
Accepted

## Context
Rover originally operated only in a reactive bug-fix mode, triggered by labeled GitHub Issues. The new requirement is to support proactive repository scanning so users can paste a GitHub repository URL and discover issues automatically.

## Decision
Add a lightweight, heuristic-based scan pipeline that clones a repository, enumerates supported source files, and applies fast, deterministic bug-pattern detection before the existing fix engine begins.

## Consequences
Positive:
- Supports proactive bug discovery without needing an existing GitHub Issue.
- Keeps the existing issue-driven fix pipeline unchanged.
- Allows the dashboard to show early findings, progress, and bug metadata.
- Makes the scan stage cheap and safe, since it does not rely on sending the entire repository to Gemini.

Negative:
- The initial scanner is not a full static analyzer and only detects a subset of bug patterns.
- Future improvements will be needed to support deeper code analysis, language semantics, and cross-file reasoning.

## Alternatives Considered
- Running Gemini over the entire repository: too expensive and unsafe for large repos.
- Using a full static analysis framework like Semgrep or SonarQube: too heavyweight for this initial architecture.

## Decision Rationale
A lightweight scanner provides the best first step: discover a core set of issues, provide structure to findings, and then reuse the existing fix flow for any selected finding. It also avoids redesigning the established `run_agent_for_issue()` engine.
